## 개요

요즘의 추세는 SSR 입니다. react 18에서 서버와 관련된 기능이 강화되면서 리액트 개발자들은 서버를 이용해 보다 나은 페이지를 구축할 수 있습니다. SSR을 지원하는 대표적인 프레임워크로 Next.js, Remix 등이 있습니다.

저는 React 18버전이 나온 이후 프론트엔드 개발을 시작했는데요. 그러다보니 서버컴포넌트, 스트리밍 등 React 18에서 신규로 추가된 서버 기능들과 기존의 SSR의 차이를 이해하는데 어려움이 있었습니다.

특히 Next.js의 App Router만을 경험했기 때문에 SSR은 서버 컴포넌트가 필수라는 인식이 박혀 관련글을 읽을수록 알고 있는 개념도 혼란스러웠던 상태를 경험했습니다. 

제 생각에는 저와 같이 개념을 구분하기 어려워하는 분들이 있을거라 생각합니다. 특히나 리액트 18 이후 리액트를 사용하는 분들은 그럴거에요. 사실 이 글을  6개월전에 작성해놨는데, 다시금 공부가 필요해 읽어보니 많은 부분이  새롭네요. 적어두길 잘한 것 같습니다. 

서론이 길었는데요. 이 글은 2부에 나눠서 작성됐습니다. 1부에서는 웹을 공부할 때 자주 만나는 렌더링의 의미에 대해서 설명합니다. 리액트를 공부할수록 개념의 혼동이 발생하는 이유가 렌더링이  갖는 다앙햔 의미 때문이라 생각하기 때문에 React 18에 대한 설명 이전에 알아두면 도움되는 지식으롯서 작성했습니다. 

2부는 리액트 18에서 새롭게 추가된 서버 기능들이 기존 리액트의 어떠한 문제점을 해결했는지에 대해 작성했습니다. 그 외에도 서버 컴포넌트의 작동 원리, next.js App Router에 보이는 Next_f이 무엇인지에 대한 설명 등 React 18과 관련된 내용도 다루고 있습니다.

## 다양한 의미로 쓰이는 렌더링(Rendering)

렌더링이란 용어는 웹 관련 개념을 설명하는데 자주 등장하는 용어 입니다. 아무래도 렌더링이라는 의미 자체가 무언가를 그리거나 현재 상태에서 다른 상태로 변화시키는 것을 의미하기 때문에 코드를 시각적으로 표현하는 웹 기술에서는 다양한 개념에서 사용되는 것은 당연한 것일 수 있습니다. 

그러므로 앞으로 렌더링이라는 일반적인 개념보다는 문맥적으로 렌더링이라는 단어가 어떠한 개념을 함축적으로 나타내는지를 파악하는게 우선입니다. 

예를 들어봅시다.

> 서버 사이드 렌더링은 서버에서 리액트를 렌더링하여 초기 HTML을 생성한 다음 클라이언트에 제공합니다. 브라우저는 서버로 부터 받은 HTML을 활용해 브라우저 렌더링을 수행합니다.

위 문장은 서버 사이드 렌더링에 대한 설명입니다. 렌더링이라는 용어가 세 번 나왔는데요. 개별 렌더링이 함축하고 있는 개념은 어느 하나와도 유사하지 않습니다. 유일한 공통점이라봤자 무언가를 생성한다는 것 말고는 없습니다. 

자 그럼 지금부터 문맥별로 렌더링이 함축하는 의미에 대해서 하나씩 이해해봅시다. 렌더링 별 의미 차이만 이해하더라도 리액트를 이해하는데 있어 한결 쉬워질거라 장담합니다. 



## 클라이언트 사이드 렌더링(CSR)과 서버 사이드 렌더링(SSR)

클라이언트 사이드 렌더링(CSR)과 서버 사이드 렌더링(SSR)에서의 렌더링은 초기 컨텐츠가 생성 되는 위치 또는 시점을 의미합니다. SSR은 사용자가 요청한 페이지가 서버에서 생성되어 클라이언트에 제공되고, 

CSR은 브라우저에서 컨텐츠를 직접 생성합니다. 그러므로 따로 서버를 유지할 필요 없습니다. 서버는 react를 실행시키는 스크립트가 담긴 비어있는 HTML만 브라우저에 제공하면, 브라우저에서 알아서 JS 코드를 실행해 컨텐츠를 생성합니다.  대신 JS를 실행시켜 컨텐츠를 생성하므로 페이지를 보기 위해선 JS 엔진이 꼭 필요해요. 

SSR은 클라이언트에서 요청한 컨텐츠가 포함된 HTML을 서버에서 생성하여 브라우저에 보냅니다. 이미 완성된 페이지를 제공하니 클라이언트 입장에서 컨텐츠를 매무 빠르게  볼 수 있어요. 그리고 이미 완성된 페이지만 제공하니 페이지 생성에 필요한 코드나 API 등을 브라우저로 보내지 않아도 되니  가볍기도 하고 보안에도 유리하죠. 



### 클라이언트 사이드 렌더링(CSR)

이제는 렌더링 별 흐름을 상세하게 이해해 봅시다.  먼저 클라이언트 사이드 렌더링(CSR) 입니다. 

1. 클라이언트가 특정 경로에 접속합니다.(ex www.example.com)

2. 서버는 클라이언트에 이미 존재하는 HTML을 전송합니다. HTML 내부에는 react 코드를 실행하는 main.js script 태그와 HTML의 기본 뼈대를 이루는 태그만 존재할 뿐 컨텐츠는 존재하지 않으므로 사용자는 비어있는 하얀색 화면을 보게 됩니다.

3. 브라우저는 HTML을 parsing 하는 과정에서 main.js가 포함된 스크립트를 실행하기 위해 코드를 다운로드 받습니다.  main.js가 포함된 스크립트는 다운로드 된 이후 바로 실행되지 않고 Dom Tree가 생성된 이후 실행됩니다.  이는 해당 스크립트의 타입이 module이기 때문에 기본적으로 defer로 처리되기 때문입니다. 

4. HTML 파싱이 완료된 다음 브라우저는 main.js를 실행해 리액트 초기 렌더링(initial rendering)을 수행합니다. 리액트 렌더링이 완료되면 실제 컨텐츠가 포함된 Dom이 생성되어 브라우저 Dom에 삽입 합니다.(리액트 초기 렌더링에 대해서는 다음 장에서 설명합니다.)

5. 브라우저 Dom이 변경되었으므로 브라우저 렌더링을 시작하여 사용자의 화면에 컨텐츠를 그려냅니다 .(브라우저 렌더링 또한 다음 장에서 설명하겠습니다.)

 

### 서버 사이드 렌더링(SSR)

클라이언트 사이드 렌더링에서 서버의 역할은 초기 HTML을 클라이언트에게 제공하는 것과 요청하는 데이터에 대한 전송 뿐입니다. 사용자에게 컨텐츠를 보여주기까지의 과정에서 서버의 개입이 최소화되죠. 

반면 서버 사이드 렌더링은 상당 부분을 서버에서 담당합니다. 대신 많은 부분을 서버로 위임한 클라이언트는 사용자와의 상호작용에 집중 합니다. 

서버 사이드 렌더링(SSR)의 렌더링 흐름을 봅시다.

1. 클라이언트가 특정 경로에 접속합니다. (ex www.example.com)

2. 서버는 리액트를 렌더링하여 컨텐츠를 포함한 초기 HTML 파일 생성하고 브라우저에 전송합니다. (리액트 렌더링은 리액트를 실행한다는 의미로 이해하고 넘어갑시다.)

3. 서버에서 제공한 HTML 안에는 CSR 처럼 클라이언트에서 리액트를 실행 시키는 스크립트가 존재합니다. 브라우저가 HTML을 Parsing하는 과정에서 해당 Script를 만나면, 브라우저 단에서의 리액트 렌더링이 시작 됩니다. 또한 서버로 부터 제공받은 HTML을 통해 브라우저 렌더링을 완료했기 때문에, 해당 HTML의 컨텐츠가 사용자에게 보여지고 있습니다.

4. 하지만 클릭, 입력 등과 같은 페이지에 대한 상호작용은 불가한 상황입니다. 상호작용을 위해선 이미 존재하는 브라우저 Dom에 이벤트 핸들러를 추가하는 절차인 Hydration을 완료해야 합니다.  Hydration은 클라이언트 단에 실행되는 리액트에 처리되며 리액트 렌더링 중 하나입니다.  따라서 SSR에서의 리액트의 렌더링은 서버에서 한 번, 브라우저에서 한 번 총 두 번 수행되는 셈입니다.
   
   > Hydration은 무엇일까요?
   > 
   > SSR의 장점 중 하나로 빠른 로딩이 있습니다. 이미 서버로부터 컨텐츠가 포함된 HTML을 받아오니 리액트에서 따로 브라우저 Dom을 생성할 필요가 없거든요.  다만 유저와 상호작용을 위해선 이미 존재하는 브라우저 Dom에 유저의 행동을 감지하는 이벤트 헨들러를 연결하는 작업은 필요합니다. 
   > 
   > 리액트는 정확한 위치에 이벤트 헨들러를 등록해야하므로 리액트에서 관리하고 있는 리액트 Dom과  브라우저 Dom이  일치하는지 우선 체크해야 합니다. 예상대로 이 둘간 차이가 없다면  이벤트 헨들러를 브라우저 Dom에 추가하고 작업을 끝냅니다.  이 작업은 브라우저 Dom을 생성하고 그에 따른 브라우저 렌더링을 유발하는 CSR 보다 훨씬 간단한 작업입니다. 
   > 
   > 만약 브라우저 Dom과 React Dom이 서로 일치하지 않는다면 이는 Hydration 에러로 리액트는 HTML로 부터 생성한 Dom을 제거하고 클라이언트 사이드 렌더링과 동일하게 초기 렌더링하여 생성한 브라우저 Dom으로 대체합니다.

5. Hydration까지 모두 마무리 되면 페이지 로딩이 완료됩니다.
   
   

서버 사이드 렌더링의 처리 흐름에서 보듯 서버 사이드 렌더링이라고 클라이언트가 아무 일도 하지 않거나 하지는 않습니다. 컨텐츠가 서버에서 생성되어 클라이언트로 전송되고 이를 활용해 리액트 렌더링을 마무리해 리액트에 의해 동작하는 페이지를 최종적으로 생성하는 것입니다.

## 리액트 렌더링

리액트를 설명하는 글에는 `리액트 렌더링을 수행한다` 또는 `리액트를 렌더한다` 라는 말을 자주 사용합니다. 이러한 용어는 리액트 내부에서 일어나는 일련의 처리 활동을 의미하는데요, 이 활동의 범위는 리액트가 브라우저의 변화를 일으키기 전까지에 해당합니다. 즉 리액트가 계산한 결과가 실제 브라우저에는 반영되기 전 단계인 것이죠. 

그러면 리액트는 어떻게 실행될까요? 공식문서는 리액트의 사이클을 Trigger - Render - Commit의 세 단계로 구분합니다. 그리고 중간 단계인 Render가 리액트에서 핵심으로 방금 앞 문단에서 말한 내용에 해당합니다. 리액트 내부에서 일련의 계산을 수행합니다. 그리고 Commit 단계에서는 Render 단계에서의 결과물을 활용해 실제 브라우저를 변경합니다. 

자 이제 개별 단계별로 어떻게 진행되는지를 설명하겠습니다.

### Trigger

트리거라는 단어는 무언가의 행동이나 행위를 유발시키는 것을 의미합니다. 리액트에서 트리거는 브라우저의 변화가 있음을 리액트에게 알리고 이를 처리해달라고 요청하는 단계 입니다. 리액트의 트리거 단계에 해당하는 상황은  1. 페이지를 초기 로딩했을때 2. 리액트의 상태(State)가 변화 되었을 때가 있습니다.

**초기 로딩 시**

사용자는 리액트로 작성된 페이지에 접속합니다. 서버는 브라우저에서 HTML을 클라이언트에 전송하는데, 이 HTML 안에는 리액트 코드를 실행시키는 Script 태그를 포함하고 있습니다. 

브라우저는 서버로부터 받은 HTML을 파싱하면서 리액트를 실행하는 코드가 들어있는 main.js 코드를 실행하게 되고, main.js 내부에 root.render()라는 매서드를 실행해서 초기 렌더링(inital rendering)을 트리거 합니다.

**상태(State) 업데이트 시**

초기 렌더링(initial rendering)이 완료된 후 사용자는 페이지의 모든 기능과 상호작용 할 수 있습니다. 버튼을 클릭한다던가 input에 값을 넣는다던가 접혀있는 아코디언을 펼치는 등의 행위를  할 수 있습니다.

사용자의 이와같은 상호작용은 컨텐츠 내용의 변화를 유발하거나 페이지 구조 변화를 유발하는 행위입니다. 버튼을 클릭해 Count를 증가시키거나 아코디언을 펼쳐 숨겨진 Dom 엘리먼트를 다시금 나타나게하는 것이 컨텐츠를 변경하거나 페이지의 구조를 변경시키는 예시입니다.

리액트는 이러한 사용자와의 상호작용에 의한 페이지 변화사항을 관리하고 이를 효율적으로 처리하도록 구현된 도구입니다. 그렇다고 페이지 내에서 발생하는 모든 변화를 리액트가 감지하고 관여하는 것은 아닙니다. 사실 리액트는 수동적인 존재로서, 페이지의 변화를 리액트에게 알려야만 작업을 처리할 수 있습니다.

물론 이러한 설계는 당연한 것입니다.유저의 상호작용이 JS에 의해서만 처리되는게 아니니까요. CSS와 HTML 만으로도 유저와의 상호작용이 가능합니다.  내용을 접고 펼칠 수 있는 아코디언은 HTML과 CSS로도 구현 가능하며, 접거나 펼치거나 하는 행위로 인한 Dom 변화는 브라우저의 렌더링 엔진이 처리합니다. (렌더링이라는 단어가 또 나왔네요. 브라우저 렌더링 엔진은 다음 장에서 설명하겠습니다.) 

리액트는 오로지 처리 요청이 들어와야만 작업을 수행하기 때문에 변화가 필요한 상황에서만 쓰면 됩니다.  아래 세가지 리액트 함수 내부에는 리액트에게 처리를 요청하는 코드가 포함되어 있습니다. 세 함수 모두 리액트에 데이터를 보관하는데 쓰이는 함수라는 공통점이 있는데요, 리액트에서 관리 중인 데이터 상태가 변화하면 그에 맞게 페이지가 변화해야하니 세 함수가 리액트에 처리를 요청하는 코드를 포함하고 있는 이유 입니다.

- useState

* createContext에서 관리하는 값의 변화
- useReducer

>  위 세 함수와 다르게 데이터를 관리하지만 리액트에게 처리를 요청하는 기능은 포함되지 않는 함수가 있습니다.  바로 useRef 입니다. useRef는 1. 데이터가 변화해도 리액트 렌더링이 필요할때와 2. 특정 Dom 엘리먼트에 직접 접근이 필요할 때 사용합니다.



### Render

Render는 리액트 내부에서 페이지 변화사항을 분석해 최적화된 업데이트 방법을 계산하는 단계입니다. 계산의 결과는 Commit 단계에서 브라우저 Dom에 업데이트 함으로써 완성됩니다. 즉 리액트 렌더링이 완료된 상태는 리액트가 어떻게 브라우저를 조작할지 청사진만을 갖고 있는 셈이에요.

Render의 모든 과정은 JS를 통해서 모두 이루어 집니다. 그래서 매우 효율적이라고 합니다. JS가 빠른 언어라서 효율적인걸까요? 당연히 아니라는건 누구나 알겁니다. 

리액트가 효율적인 이유는 어떻게 브라우저를 업데이트 해야할지 청사진을 가지고 한 번의 브라우저 렌더링만으로 페이지를 업데이트하기 때문입니다.  

만약 리액트가 청사진 없이 Dom 조작 코드를 만날때마다 Dom을 조작한다고 생각해보세요. 그럴때마다 매번 브라우저 렌더링이 수행되는 겁니다. 브라우저 렌더링은 일련의 복잡한 처리과정을 수행해야므로 많은 컴퓨팅 리소스를 필요로 합니다. 그에 반해 JS 객체인 리액트 엘리먼트를 업데이트 하는 것은 일도 아니니 리액트가 효율적이라는 겁니다.

##### 초기 렌더링(Initial Rendering)

초기 렌더링은 CSR과 SSR에 따라 다르게 진행됩니다. 

**CSR**

CSR에서의 초기 렌더링은 [createRoot(domNode, options)](https://ko.react.dev/reference/react-dom/client/createRoot)을 사용합니다. 패러미터인 domNode는 실제 브라우저 Dom 노드를 넣어야 합니다. 아래 코드에서 domNode 변수는 Id가 root인 엘리먼트를 의미하고 domNode는 createRoot의 인자로 들어갑니다. 

createRoot으로 생성된 root 객체는 render 메서드와 unmount 메서드를 갖고 있습니다. `root.render`는 parameter로 reactNode를 받는데요. React Node를 실행하면 UI를 그리기 위한 각종 데이터가 포함된 JS 객체(Object)를 반환합니다. 그리고 이러한 객체를 리액트 엘리먼트(React Element)라 부릅니다. 리액트 엘리먼트는 children이라는 key 있는데, 하위 React Node는 상위 리액트 Node의 자식으로 들어가 있습니다. 그래서 리액트 엘리먼트는 트리 구조를 갖습니다.

예시

리액트는 리액트 엘리먼트를 활용해 기존 UI 데이터와 업데이트 된 UI 데이터를 브라우저 Dom에서 어느 부분을 업데이트할지 계산하는데요.  초기 렌더링의 경우 기존 UI가 없으니 리액트 엘리먼트를 생성하고  브라우저 Dom을 그립니다.

**SSR**

SSR은 서버와 클라이언트에서 별도로 Render를 수행합니다. 리액트로 작성된 페이지에 접속하면 서버는 페이지 컨텐츠 생성을 위해 리엑트를 렌더합니다. 서버에서 리액트를 렌더하면 컨텐츠가 들어있는 HTML 이 생성되어 브라우저에게 제공됩니다. (서버 컴포넌트와 관련된 내용은 리액트 18을 설명하는 글에서 다루겠습니다.)

서버에서 Render는 [renderToPipableStream(reactNode, options)](https://ko.react.dev/reference/react-dom/server/renderToPipeableStream#rendertopipeablestream)를 사용합니다. 패러미터인 reactNode는 서버에서 렌더링하려는 reactNode 입니다. 해당 함수에 쓰이는 옵션은 bootstrapScript, onShellReady 외에도 다양한데요. 주로 이 둘을 기본값으로 사용하므로 이에 대해서 설명하겠습니다. 그 외 다른 옵션은 공식문서에 설명하고 있으니 링크를 참고하시기 바랍니다.

```jsx
import Suspense from 'react'
import { renderToPipeableStream } from 'react-dom/server';

const { pipe } = renderToPipeableStream(<App />, {
  bootstrapScripts: ['/main.js'],
  onShellReady() {
      // shell 렌더링 완료 후 실행 될 callback 함수
    response.setHeader('content-type', 'text/html');
    pipe(response);
  }
});

export default function App() {
  return (
    <html>
      <head>
        <meta charSet="utf-8" />
        <meta name="viewport" content="width=device-width, initial-scale=1" />
        <link rel="stylesheet" href="/styles.css"></link>
        <title>My app</title>
      </head>
      <body>
        <div>Contents...</div>
        <Suspense fallback={<div>Loading....</div>}>    
            <Router />
        </Suspense>    
        <div>Contents...</div>
      </body>
    </html>
  );
}

---
서버로부터 받은 HTML
<!DOCTYPE html>
<html>
  <!-- ... 컴포넌트의 HTML ... -->
</html>
<script src="/main.js" async=""></script>
```

`bootstrapScripts`는 클라이언트에서 리액트 렌더를 실행시키기 위한 경로를 설정하는 옵션입니다. 옵션 명에서 bootstrap이라는 단어가 포함되어있는데, 이는  알아서 스스로 시작한다는 의미가 있습니다. bootstrapScripts에 리액트 렌더를 실행하는 코드가 들어있는 js 파일명을 작성하면, 리액트 서버 렌더로 생성한 HTML 하단에 `<script src="/main.js" async=""></script>`를 삽입합니다. 

아래 예시코드 하단의 HTML을 보면 해당 script가 존재하는 것을 확인할 수 있습니다.  브라우저는 HTML을 파싱하는 과정에서 main.js를 실행하게 되고 그러면 클라이언트단에서의 렌더가 시작됩니다.



다음으로 `onShellReady`는 Shell 영역의 렌더가 완료된 다음에 실행시키는 Callback 함수를 정의합니다.  Shell이라 함은 Suspense 경계 외부 영역을 지칭하는데요. 예시 코드에서 Suspense 영역 밖의 모든 영역(suspense부터 html 태그까지)이 Shell이 됩니다. Shell 영역에 속한 리액트 노드가 HTML로 변환되고나면 onShellReady에서 정의한 함수를 실행해 Shell에 해당하는 HTML을 브라우저로 스트림합니다. 



다음으로 클라이언트의 리액트 렌더에 대해 설명하겠습니다. 앞에서 서버는 클라이언트에서 리액트를 실행시키는 bootstrap script를 포함해 HTML을 생성한다고 했는데요. 클라이언트는  main.js 내부의 hydrateRoot를 실행해 초기 렌더링을 시작합니다.

```jsx
import { hydrateRoot } from 'react-dom/client';

const domNode = document.getElementById('root'); // 실제 dom Node
hydrateRoot(domNode, reactNode);


```

[hydrateRoot(domNode, reactNode, options)](https://ko.react.dev/reference/react-dom/client/hydrateRoot)의 reactNode는 서버에서 HTML을 생성하기 위해 사용했던 React Node와 동일해야 합니다. 이벤트 핸들러를 브라우저 Dom에 더하는 작업을 수행하려면 리액트에서 관리하는 UI 데이터와 실제 브라우저 Dom이 일치해야합니다. 불일치 할 경우 Hydration 에러로 처리하여  이미 존재하는 브라우저 Dom을 제거하고 CSR과 동일한 렌더링을 통해 새로운 브라우저 Dom을 생성합니다.

##### State 업데이트에 의한 리렌더(re-render)

초기 렌더링이 완료되면 사용자는 브라우저와 상호작용이 가능합니다. 브라우저의 상호작용에서 리액틍의 set 함수를 포함하여 처리하면 리액트의 상태가 변화하고 그에 따른 리렌더를 수행합니다.

상태 변화에 의한 업데이트는 초기 렌더링과 다르게 이미 리액트가 현재 페이지에 대한 UI 데이터(`currentTree`)를 데이터를 관리하고 있으므로, 업데이트에 의해 새로 생성되는 UI 데이터(`workingInProgess`Tree)를 비교하여 어느 부분이 업데이트가 필요한지, 어느 부분은 업데이트가 필요 없는지를 판단합니다. 

리액트가 현재 페이지에 대한 UI 데이터(`currentTree`)를 데이터를 관리하고 있으므로, 업데이트에 의해 새로 생성되는 UI 데이터(`workingInProgess`Tree)를 비교하는지에 대한 상세 절차는  [How does React re-render internally?](https://jser.dev/2023-07-18-how-react-rerenders/#how-react-re-renders-internally)에 그림으로 자세히 나와있으니 이를 참고 바랍니다.

### Commit

커밋은 리액트의 랜더 결과물이라 할 수 있는 UI 청사진을 활용해 실제 브라우저의 Dom을 조작하는 단계입니다. Render 단계에서도 설명했듯 코드를 읽어내려가면서 처리하는 Dom 조작이 브라우저 렌더링을 매번 유발하는 것은 상당히 비효율 적입니다. 리액트는 Dom 조작을 한 번만 수행할 수 있도록 리액트 내부에서 업데이트를 관리하고 최적화 합니다.  이제는 이 훌륭한 결과물을 활용해 브라우저를 업데이트 하기만 하면 됩니다.

##### 초기 렌더링

초기 렌더링은 컨텐츠에 대한 브라우저 Dom이 존재하지 않으므로 모든 Dom을 생성합니다.  리액트가 브라우저 Dom을 생성하는 방법은 [How does React do the initial mount internally?](https://jser.dev/2023-07-14-initial-mount/#how-react-does-initial-mount-first-time-render-) 에 잘 나와있습니다. 이렇게 생성한 Dom은  web API인 AppendChild를 사용해 `id='root'` 인 Dom 엘리먼트 내부에 삽입합니다. 브라우저 Dom의 변화가 감지되었으므로 브라우저 엔진은 브라우저 렌더링을 절차를 수행하여 최종적인 페이지를 그려냅니다.

##### 상태 업데이트에 의한 커밋

리액트는 이전 상태의 UI 데이터와 업데이트 된 UI 데이터를 비교해 브라우저 Dom의 변화사항을 기록하는 청사진을 갖고 있습니다. 커밋 단계에서는 이러한 청사진을 활용해 전체 Dom을 새로 만들 필요 없이 변경된 부분만 핀포인트로 업데이트 합니다.

## 브라우저 렌더링

브라우저는 여러 종류의 모듈로 구성되어 있습니다. 모듈에는 크롬 기준 JS를 해석하고 실행하는 V8, HTML 파싱해 Dom 트리를 생성하는 Blink, CSS를 해석해 CSSOM 트리를 생성하는 ... 실제 화면에 컨텐츠를 그리는 UI Backend 등이 있습니다. 

브라우저 렌더링은 서버로부터 받아온 HTML을 읽어 브라우저 내 다양한 모듈을 활용해 화면에 컨텐츠를 그리는 전체의 과정을 말합니다. 브라우저 내부에서 일어나는 작업은 상당히 복잡하며 상당한 리소스를 필요로 합니다. 심지어 GPU까지 활용합니다.

브라우저 렌더링의 세부 절차는 아래 그림과 같이 순서적으로 진행됩니다. 리액트 렌더링은 JS 엔진에서 처리되므로 브라우저 렌더링의 세부 절차 중 하나입니다. 리액트가 render 결과를 commit 하면 브라우저는 리액트에게 요청받은 대로 브라우저 내 여러 모듈을 활용해  변경사항을  화면에 그리게 됩니다.

이 장에서는 브라우저 렌더링의 세부 절차 보다는 리액트를 사용하는 페이지를 어떻게 브라우저 렌더링이 되는지에 대해 설명하도록 하겠습니다. 브라우저 렌더링 절차에 대한 상세한 내용이 궁금하시거나 배경 지식이 필요하신 분은 아래 공유한 링크를 활용하시기 바랍니다. 



##### 초기 렌더링

브라우저 렌더링은 서버로부터 HTML 파일을 요청하는 것에서부터 시작합니다. 이때 접속하고자 하는 페이지가 CSR로 구성된 경우라면 브라우저에 비어있는 HTML 파일을 제공합니다. SSR로 구성된 경우라면  컨텐츠가 포함된 HTML을 브라우저에게 제공합니다.

브라우저는 서버로부터 HTML에 대한 첫 패킷이 도착하면 HTML 파싱을 시작합니다. 통신 간 기본적으로 14kb 단위로 패킷을 보내는데, 브라우저는 첫번째 패킷이 도착하면 바로 파싱을 수행하는 것이죠. 다르게 말해  HTML 전체를 받고나서야 파싱을 시작하는 것은 아니라는 겁니다. 그래서 웹 페이지 성능을 측정하는 요소중에 서버로부터 첫 패킷이 도달하는 시간을 의미하는 Time to first byte (TTFB)가 있습니다.  TTFB가 빠를수록 브라우저는 페이지 로드 속도를 단축시킬 수 있겠죠.



서버로부터 받은 HTML은 클라이언트에서 리액트를 실행시키는 스크립트 태그를 포함하고 있습니다. HTML을 읽어 Dom Tree를 만드는 렌더링 엔진은 파싱 중 스크립트 태그를 만나면 파싱을 중단하고 JS 처리 결과를 기다려야 합니다. 그러니 리액트 렌더링이 길어질수록 사용자는 처음 페이지에 접속하고나서 흰화면을 바라보는 시간이 길어지게 됩니다. 

이는 CSR 뿐만 아니라 SSR도 마찬가지입니다. 대신 SSR은 리액트를 실행해 컨텐츠를 생성해야 하는 절차가 없기 때문에 매우 빠르게 화면이 나타나며, hydration을 위해서 Dom Tree가 필요하므로 Dom Tree 생성 이후 리액트가 실행되도록 하는 로직이 작성되어 있습니다.  

First contentful paint(FCP)는 사용자가 처음으로 컨텐츠를 접하는 시간을 의미하는 웹 페이지 성능 측정 요소입니다. FCP 시간을 최소화하는 것이 유저 경험과 SEO에서 필수이므로 최신 웹 기술은 Dom Tree 생성을 막는 요소를 최대한 줄여나가는데 집중하고 있습니다. 그 중 가장 큰 요소인 JS를 최소화하기 위해 번들 사이즈를 최소화하여 다운로드를 빠르게 하거나, 아예 CSR에서 SSR로의 변화로 이어지고 있습니다.



CSR의 초기 브라우저 렌더링은 다음의 절차로 수행됩니다. HTML 파싱 중 리액트 코드가 포함된 스크립트를 만나면 서버에게 해당 코드를 요청합니다. 파일명은 주로 Index.js 또는 main.js 입니다. 스크립트 설정이 type='module'이므로 기본 처리가 defer 입니다. 그러므로 Dom Tree가 생성 된 후 JS 코드를 실행합니다. JS 처리가 완료될 때까지 브라우저 렌더링은 중단됩니다. 사용자는 JS 처리가 끝나고 렌더링이 완료될 때까지 흰 화면만 바라봅니다.

JS는 index.js 내 `createRoot`을 실행하여 root 객체를 생성하고 `root.render`를 실행해 리액트 렌더링을 시작합니다. 리액트는 리액트 컴포넌트를 처리하여 컨텐츠에 필요한 UI 데이터를 생성합니다. 

커밋 단계에서 리액트는 렌더 단계에서의 결과물인 UI 데이터를 활용해 브라우저 Dom Element 생성을 요청하고 appendChild를 요청합니다. 렌더링 엔진(크롬의 경우 blink)이 이러한 요청을 직접 처리합니다.

지금까지의 프로세스로 Dom Tree 생성은 완료되었으나 브라우저 렌더링은 이제 시작입니다. CSSOM을 생성해 Dom tree와 합성하여 Render Tree를 만들고 이를 기반으로 Layout을 계산하고 최종적으로 Paint하여 컨텐츠를 화면에 나타냅니다.



SSR의 초기 브라우저 렌더링은 사용중인 프레임워크에 따라 세부적인 절차가 다르므로 개괄적인 흐름이 이렇다 정도로 이해하시면 좋겠습니다.

SSR은 서버에서 컨텐츠를 이미 생성하여 클라이언트로 제공하기 때문에 클라이언트에서 직접 컨텐츠를 생성할 필요가 없습니다. 대신 Hydration이라는 클라이언트 단의 작업을 통해 서버의 컨텐츠를 리액트가 관리할 수 있도록 처리 해야합니다. 

Hydration은 이미 완성된 브라우저 Dom Element에 이벤트 리스너를 추가하는 절차입니다. 이 절차를 수행하기 위해서는 완성된 브라우저 Dom Tree가 필요합니다. 그러므로 클라이언트에서는 Dom Tree 생성이 완료된 이후 hydatreRoot을 수행할 수 있도록 로직을 작성해야 합니다. 

이를 위한 가장 간단한 방법은 index.js 스크립트를 HTML 최하단에 위치시키는 것입니다. 최하단에 위치시키면 JS 다운로드를 다운로드 받는 중에  Dom Tree가 생성이 완료되고 JS가 실행되는 흐름을 가질 수 있습니다. 물론 프레임워크는 hydrate를 간단한 방법으로 수행하지 않습니다. 프레임워크는 번들러와의 조합을 통해 화면에 컨텐츠가 그려지고 난 다음 Hydrate 수행할 수 있도록 설정합니다.



##### State 업데이트에 의한 리렌더(re-render)

사용자의 행동의 결과 상태(state)가 업데이트 되면 리액트는 render 단계를 수행해 업데이트가 필요한 UI Data를 생성하고, 이를 활용해 Commit 단계에서 브라우저 Dom에 대한 실제 업데이트를 요청합니다.  

브라우저 Dom의 변화가 생기면 브라우저 렌더링은 다시금 수행되는데, 이떄는 Dom Tree, CSSOM Tree, Render Tree를 생성하는 단계를 건너뛰고 Layout 단계 또는 paint 단계부터 시작합니다.  

이는 상태 변화가 화면의 어떠한 변화를 야기하느냐에 따라 다릅니다. 가령 margin, Padding, height, width와 같이 layout을 건드리는 경우라면 Layout 단계서부터 시작하며 이를 Reflow라 부릅니다. 반면 bg-color, box-shadow와 같은 기존의 layout을 다시금 사용하더라도 변화를 처리할 수 있다면 paint 단계서부터 시작하며 이를 Repaint라 합니다.

### 마무리

컴퓨터 분야를 공부하면서 매번 느끼는 점은 하나의 용어를 다양한 상황 다양한 개념을 설명하는데 사용한다는 것입니다. 제게는 웹 분야를 공부하면서  '렌더링' 이라는 단어가 그러했습니다. 렌더링은 어느 개념에서나 존재하고 다양한 단어와 조합되어 쓰이곤 하다보니 알다가도 뒤돌면 사람을 혼란스럽게 만듭니다. 아무래도 코드를 작성할 때 변수명을 작성하는게 어렵듯, 개념을 설명할 대상을 부르는 용어를 새로 만든는게 어렵지 않기 때문에 그런가 싶습니다. 렌더링 외에는 딱 맞는 용어가 없을 수 도 있구요

아무튼, 이 글을 작성하고나니 이제는 렌더링에 대해 혼라스러워 할 일이 줄어들 것 같습니다. 그리고 렌더링의 의미 차이를 세세하게 파다보니  덕분에 브라우저 작동원리를 큰 그림으로 이해할 수 있기도 했구요. 

얼마나 많은 분들이 이 글을 끝까지 읽으실지는 모르겠지만, 제 글이 렌더링이 갖고 있는 다양한 의미를 이해하는데 도움 되셨으면 좋겠습니다. 그리고 긴 글을 끝까지 읽어주셔서 감사합니다.
